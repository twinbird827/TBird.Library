## 1. プロジェクト概要

### 1.1 背景
楽天ウェブサービスは 2026 年新仕様（`openapi.rakuten.co.jp` ドメインへ移行、`applicationId + accessKey` 二重認証、Referer/Origin ヘッダ必須化、ドメイン許可制）へ移行した（告知 2026-02-10、移行期間〜5/13、5/14 旧ドメイン停止）。Android ネイティブアプリは HTTP クライアントとして `Referer`/`Origin` ヘッダ自体は付与できるが、楽天が照合する**固定の Web ドメイン（「許可されたWebサイト」に登録する出所）を持たず**、また**配布物に `applicationId`/`accessKey` を埋め込むのは避けたい**（逆コンパイルで漏洩する）。このため Android から楽天 API を直接利用する構成は現実的でない。

そこで、自宅サーバー（Windows + IIS、MyDNS.JP の DDNS で `kaz.server-on.net` に到達）に薄い HTTPS リバースプロキシを立て、

1. Android アプリ → 中継サーバー（共有シークレットで認証）
2. 中継サーバー → 楽天 API（登録ドメイン `kaz.server-on.net` を Referer/Origin に付与）

という構成で楽天 API を利用する。

### 1.2 中継サーバーの責務
- Android アプリからの HTTPS リクエストを受ける（共有シークレット認証）
- リクエスト内容に応じて楽天 Kobo の指定 API へ転送
- 転送時に `applicationId`, `accessKey` をクエリに付与（§3.2.3。公式上 accessKey はヘッダでも可だが本サーバーはクエリで統一）、`Referer` / `Origin` ヘッダを `https://kaz.server-on.net/` で付与
- 楽天 API のレスポンスを**そのまま透過**で返す（ステータス・本文・主要ヘッダを保持。`Set-Cookie` 等のクライアント不要ヘッダは除外）
- レート制限の二重防御（クライアント側に加え、サーバーでも上限を設け 429 を返す）
- 認証失敗・楽天 API エラー・タイムアウト等のログ記録

### 1.3 やらないこと
- 楽天 API レスポンスの整形・型付け・キャッシュ・集約（透過プロキシに徹する）
- 認可（複数ユーザーの管理）。共有シークレットによる単一クライアント前提
- 中継サーバーが他の楽天 API（市場・トラベル等）を提供すること。**Kobo の指定 API のみ**

### 1.4 利用環境
| 項目 | 内容 |
|---|---|
| サーバー OS | Windows（10/11 / Server）+ IIS |
| ドメイン | `kaz.server-on.net`（MyDNS.JP の DDNS で運用） |
| ポート公開 | ルーター側ポートフォワーディング `外部 49443 → 内部 443` |
| 外部からの URL | `https://kaz.server-on.net:49443/` |
| HTTPS 証明書 | Let's Encrypt（win-acme で取得・自動更新） |
| 想定クライアント | Android 新刊チェッカー（個人用、1端末） |

## 3. 機能要件

### 3.1 機能一覧
| 機能ID | 機能名 | 概要 |
|---|---|---|
| R-001 | 共有シークレット認証 | 全リクエストで `X-Relay-Auth` ヘッダを検証 |
| R-002 | Kobo 電子書籍検索プロキシ | `POST /api/kobo/search` → 楽天 Kobo 電子書籍検索 API へ透過転送 |
| R-003 | Kobo ジャンル検索プロキシ | `POST /api/kobo/genres` → 楽天 Kobo ジャンル検索 API へ透過転送 |
| R-004 | レート制限（受信側） | クライアント IP 単位で毎秒上限を設け、超過時 429 を返す |
| R-005 | ヘルスチェック | `GET /healthz` → 200 OK（IIS 死活監視用、認証不要） |
| R-006 | ログ出力 | リクエスト受信・楽天 API 応答・エラーをログに記録 |

### 3.2 中核ロジック：透過プロキシ

#### 3.2.1 リクエスト処理フロー
1. **受信**: Android から HTTPS リクエスト（`POST /api/kobo/search` または `/api/kobo/genres`）。本文は楽天 API に渡したいパラメータの JSON（後述 3.2.5）
2. **レート制限**: クライアント IP 単位の上限を確認。超過なら 429 を返し処理打ち切り（**認証より前**に評価する。`中継サーバー_CLAUDE.md` §6.5 のミドルウェア順 `UseRateLimiter` → 認証ミドルウェア と一致させ、未認証リクエストの氾濫・共有シークレット総当たりを入口で抑えるため）
3. **認証**: `X-Relay-Auth` ヘッダの値を、サーバー保持の共有シークレットと**定数時間比較**で照合。不一致なら 401 を返し処理打ち切り
4. **クエリ組み立て**: リクエスト本文の JSON をクエリ文字列に展開し、サーバー側で保持する `applicationId` と `accessKey` を付与。**各値は個別に URL エンコード**（`Uri.EscapeDataString`）すること。特に `sort` の `+releaseDate` は `%2BreleaseDate` にしないと上流で `+` が空白へ変換されソート指定が壊れる（楽天公式: `keyword`/`title`/`author`/`publisherName`/`sort` は UTF-8 で URL エンコード必須。URL 全体ではなく**値部分のみ**をエンコード）。JSON 値が数値（例 `hits:30`, `formatVersion:2`）の場合は文字列化してから付与。クライアント本文に万一 `applicationId`/`accessKey`/`affiliateId` が含まれていても**無視し、必ずサーバー保持値で上書き**する（クライアントからの認証情報注入・取り違えを防ぐ）
5. **転送**: 楽天 API へ `GET` リクエスト。タイムアウト 15 秒（実現方法は §7：`HttpClient.Timeout` ではなく `HttpContext.RequestAborted` と 15 秒の `CancelAfter` をリンクした `CancellationToken` を使い、**クライアント切断（→そのまま伝播）と上流タイムアウト（→504）を区別**する）。ヘッダに `Referer: https://kaz.server-on.net/`, `Origin: https://kaz.server-on.net`, `User-Agent: NewReleaseChecker-Relay/1.0` を付与
6. **応答転送**: 楽天 API のステータスコード・`Content-Type` ヘッダ・本文をそのままクライアントに返す（後述 3.2.4 で除外するヘッダを明記）
7. **ログ**: 受信時刻・エンドポイント・楽天 API ステータス・所要時間を記録

#### 3.2.2 認証（R-001）
- **方式**: 共有シークレット。HTTP ヘッダ `X-Relay-Auth: <secret>` で送受信
- **保管**: サーバー側は `appsettings.Secrets.json`（`.gitignore` 除外）に保持。Android 側は `Secrets` クラスに `RelayServerApiKey` として保持
- **比較**: `CryptographicOperations.FixedTimeEquals` でタイミング攻撃対策
- **失敗時**: ステータス 401、本文は最小限（`{"error":"unauthorized"}`）。詳細を返さない
- **適用範囲**: `/api/kobo/*` の全エンドポイント。`/healthz` は認証不要

#### 3.2.3 上流ヘッダ付与（楽天 API 認証通過の核心）
中継サーバーから楽天 API への送信時、以下を必ず付与する。

| ヘッダ | 値 |
|---|---|
| `Referer` | `https://kaz.server-on.net/`（末尾スラッシュあり） |
| `Origin` | `https://kaz.server-on.net`（末尾スラッシュなし） |
| `User-Agent` | `NewReleaseChecker-Relay/1.0`（任意の文字列で可。空白送信は避ける） |
| `Accept` | `application/json` |

`applicationId` は**クエリパラメータ必須**。`accessKey` は楽天公式仕様上**ヘッダ（`accessKey: <値>`）・クエリのどちらでも可**だが、本サーバーは**クエリ付与で統一**する。両キーは 2026 年新仕様で**両方必須**（片方欠落で 400 `wrong_parameter`「accessKey must be present」等）。値は配置時に楽天アプリ管理画面で再確認すること（→ §8）。

> 📌 **検証メモ（2026-06 公式ドキュメント確認済）**: 楽天 Kobo 電子書籍検索 API の新エンドポイントは `https://openapi.rakuten.co.jp/services/api/Kobo/EbookSearch/20170426`。2026 年移行でドメインは `app.rakuten.co.jp`→`openapi.rakuten.co.jp` に変わったが、**Kobo はパス `/services/api/Kobo/...` を据え置き**（楽天市場 API が `/services/api/` →`/ichibams/api/` とパス変更したのとは異なり、Kobo にパス変更は無い）。**ただしバージョン番号は API ごとに異なる**——電子書籍検索は `20170426`、**ジャンル検索は `20131010`**（公式タイトル「楽天Koboジャンル検索API (version:2013-10-10)」・リクエストURL `.../Kobo/GenreSearch/20131010` で確認）。電子書籍検索は旧 Android 実装の `app.rakuten.co.jp/services/api/Kobo/EbookSearch/20170426` を**ドメイン差し替えのみで流用可**だが、**ジャンル検索は旧実装（`RakutenKoboApiClient.cs`）が誤って `GenreSearch/20170426` を使っているため、ドメインに加えてバージョンも `20131010` へ直す必要がある**（旧定数をそのまま流用しないこと）。

### 3.2.4 透過するヘッダ・本文の扱い
楽天 API のレスポンスをクライアントに返す際、以下の方針（**許可リスト方式**＝転送するヘッダを明示列挙し、それ以外は転送しない。§7 と整合）:
- **そのまま返す**: ステータスコード、`Content-Type`、本文バイト列
- **本文を無加工で透過するため `HttpClientHandler.AutomaticDecompression` は無効のまま**にし、上流が `Content-Encoding`（gzip 等）を付けた場合はその本文と `Content-Encoding` ヘッダをセットで転送する（中継側で展開・再圧縮しない）。`Content-Length` は ASP.NET Core / IIS に再計算させる
- **除外する**: `Set-Cookie`, `Transfer-Encoding`, `Connection`, `Server`（サーバーが自動付与）、`Date`（IIS が自動付与）。これらは中継しない
- **追加する**: 必要に応じて `X-Relay-Elapsed-Ms`（中継処理時間。デバッグ用）

### 3.2.5 エンドポイント仕様

#### R-002: `POST /api/kobo/search`
**用途**: 楽天 Kobo 電子書籍検索 API への透過プロキシ。Android の F-001（検索）/ F-003/F-004（新刊チェック）/ F-009（発売予定表）/ F-011（ランキング）が使用。

**リクエスト**:
```http
POST /api/kobo/search HTTP/1.1
Host: kaz.server-on.net:49443
X-Relay-Auth: <共有シークレット>
Content-Type: application/json

{
  "keyword": "薬屋のひとりごと",
  "hits": 30,
  "sort": "+releaseDate",
  "format": "json",
  "formatVersion": 2
}
```
- JSON のキー名・値は楽天 Kobo 電子書籍検索 API のクエリパラメータ名・値とそのまま 1:1 対応
- `applicationId` / `accessKey` はクライアントから送らない（サーバーで付与）

**レスポンス**: 楽天 API のレスポンスをそのまま透過。

#### R-003: `POST /api/kobo/genres`
**用途**: 楽天 Kobo ジャンル検索 API への透過プロキシ。Android の F-009/F-011 のジャンルメニュー生成で使用。

**リクエスト**:
```http
POST /api/kobo/genres HTTP/1.1
Host: kaz.server-on.net:49443
X-Relay-Auth: <共有シークレット>
Content-Type: application/json

{
  "koboGenreId": "000",
  "format": "json",
  "formatVersion": 2
}
```

**レスポンス**: 楽天 API のレスポンスをそのまま透過。

#### R-005: `GET /healthz`
**用途**: IIS の死活監視・動作確認。
**認証**: 不要。
**レスポンス**: 200 OK、本文 `{"status":"ok"}`。

### 3.3 レート制限（R-004）
- **方式**: ASP.NET Core 標準 `RateLimiter`（`AddRateLimiter` + `FixedWindowLimiter` または `TokenBucketLimiter`）
- **粒度**: クライアント IP 単位（`HttpContext.Connection.RemoteIpAddress`）。個人用なので IP ホワイトリスト的にも機能する
- **上限値（初期）**: **1秒あたり 2 リクエスト**まで（Android 側で 1 シリーズ 1 秒間隔の `SiteRateLimiter` が動くため、通常はサーバー側で発火しない。バグ・暴発時の保険として設定）。`appsettings.json` で変更可能
- **超過時**: 429 Too Many Requests、`Retry-After: 1` ヘッダを返す
- **`/healthz` は対象外**

> ⚠️ **上流（楽天 API）側のレート制限に注意**: 2026 年新仕様は旧来より厳しく、コミュニティ実測で**リクエスト間隔 1.5 秒未満だと上流が 429（`too_many_requests`）を返す**事例が報告されている。中継サーバーの上記上限（0.5 秒間隔相当）は「クライアント暴発の粗い保険」であって**上流 429 を防ぐものではない**。上流 429 は §4.4 のとおり**そのまま透過**してクライアントに返す。実運用での 429 多発を防ぐ主たるペース制御は **Android 側 `SiteRateLimiter` の間隔を 1 秒→1.5 秒以上に引き上げる**ことで対応する（`Android引き継ぎメモ.md` 側で調整。本サーバーは透過に徹し上流ペーシングは持たない）。

### 3.4 ログ要件（R-006）
- **ライブラリ**: ASP.NET Core 標準 `ILogger<T>`（`Microsoft.Extensions.Logging`）
- **出力先**: 標準出力＋ファイル（後述 CLAUDE.md で実装詳細）
- **記録イベント**:
  - **Information**: リクエスト受信（エンドポイント・所要時間・楽天 API ステータス）
  - **Warning**: 共有シークレット認証失敗（IP・タイムスタンプのみ。送信値は記録しない＝ログ汚染防止）、レート制限超過
  - **Error**: 楽天 API への接続失敗・タイムアウト・5xx 応答、想定外の例外
- **個人情報**: クエリパラメータ（検索キーワード等）はログに含めて問題なし（個人用なので）。ただし `accessKey`・`applicationId`・`X-Relay-Auth` の値は**絶対にログに出さない**

## 4. 非機能要件

### 4.1 パフォーマンス
| 項目 | 要件値 |
|---|---|
| 中継処理オーバーヘッド | 50ms 以内（楽天 API への通信時間を除く） |
| 楽天 API へのタイムアウト | 15 秒 |
| 並行リクエスト | 個人用・1端末想定のため特別な配慮なし（ASP.NET Core のデフォルト） |

### 4.2 起動・終了
- **起動**: IIS が必要に応じてアプリプールを起動。設定読込・DI 構築・楽天 API への HttpClient 準備
- **アイドル時の挙動**: IIS のアプリプール設定でアイドルタイムアウトを長めに設定（IIS デプロイ手順は CLAUDE.md 参照）
- **終了**: IIS 任せ（特別な後処理不要）

### 4.3 秘密情報管理
- 楽天 `applicationId`、楽天 `accessKey`、共有シークレット は `appsettings.Secrets.json` に保持
- このファイルは **`.gitignore` で Git 管理から除外**
- リポジトリには `appsettings.Secrets.json.example`（ダミー値）のみコミット
- 値はサーバーへの配置時に手動で設置
- 設定読み込みは ASP.NET Core 標準の `Configuration` 機構（`builder.Configuration.AddJsonFile("appsettings.Secrets.json", optional: false)`）

### 4.4 エラー応答
| 状況 | ステータス | 本文 |
|---|---|---|
| 認証失敗 | 401 | `{"error":"unauthorized"}` |
| レート制限超過 | 429 | `{"error":"rate_limited"}` |
| 楽天 API タイムアウト | 504 | `{"error":"upstream_timeout"}` |
| 楽天 API 接続失敗（DNS / TLS / ネット断） | 502 | `{"error":"upstream_unreachable"}` |
| 楽天 API が 4xx/5xx を返した | **そのままのステータスと本文を透過** | （楽天のエラー本文） |
| 想定外の例外 | 500 | `{"error":"internal"}` |

### 4.5 配布・デプロイ
- ビルド: `dotnet publish -c Release -r win-x64 --self-contained false`（ASP.NET Core Hosting Bundle が IIS に入っている前提）
- 配置: IIS のサイトディレクトリにコピー。アプリプールは「No Managed Code」（in-process ホスティング）

> ⚠️ **2026 年新仕様での ID 取得に関する注意**: 新仕様では `applicationId` の形式が **UUID 形式**（例 `e5e2671a-b454-...`）に変わり、`accessKey` の発行が必須化された。**2026 年 2 月移行より前に発行した旧 ID（数字のみ）は利用不可**で、アプリの**新規（再）登録**が必要。既存 Android 実装（`Secrets.cs`）に旧形式 ID が入っている場合はそのままでは通らないため、中継サーバー用に取得し直した `applicationId`(UUID) + `accessKey` を使うこと。

## 7. 実装上の注意事項・設計方針

- **透過プロキシに徹する**: レスポンスの整形・キャッシュ・集約はしない（楽天 API の生レスポンスを Android 側で扱う既存設計を維持するため）
- **共有シークレット比較は定数時間で**: `CryptographicOperations.FixedTimeEquals` を使用。`==` や `string.Equals` でタイミング攻撃の余地を残さない
- **シークレットをログに出さない**: 認証失敗ログでも送信値を記録しない（ログ自体が漏れた場合の二次被害防止）
- **`HttpClient` は `IHttpClientFactory` 経由で取得**: ソケット枯渇・DNS キャッシュ問題を回避する標準パターン。**型付き HttpClient（`AddHttpClient<IRakutenProxy, RakutenProxyService>`）を Singleton 登録しない**こと——型付きクライアントを Singleton にすると `HttpMessageHandler` がローテートされず、まさにここで避けたい DNS 滞留が起きる。プロキシは型付きクライアント（既定 Transient）として登録するか、Singleton にするなら `IHttpClientFactory.CreateClient()` をリクエスト毎に呼ぶ（→ `中継サーバー_CLAUDE.md` §4/§6.5 の DI 記述はこの方針で統一する）
- **タイムアウトは `HttpClient.Timeout` ではなく `CancellationToken` で**: `CancellationTokenSource.CreateLinkedTokenSource(HttpContext.RequestAborted)` に `CancelAfter(TimeSpan.FromSeconds(15))` を組み合わせ、`SendAsync` にリンクトークンを渡す。例外時は `RequestAborted.IsCancellationRequested` で**クライアント切断（伝播）と上流タイムアウト（504）を区別**する（`HttpClient.Timeout` を併用すると両者が同じ `TaskCanceledException` になり区別できないため設定しない）
- **ヘッダの許可リスト方式**: 透過するのは原則 `Content-Type`（および無効化していない `Content-Encoding`）のみを明示的に列挙し、それ以外は転送しない。除外したいヘッダ（§3.2.4）を確実にフィルタ
- **設定値は `IOptions<T>` パターンで注入**: `RakutenOptions`, `RelayAuthOptions` を強く型付け

### 実装時の検証事項
1. ~~正確なベース URL・パス・APIバージョン~~ → **解決済（2026-06 公式）**: 電子書籍検索 `https://openapi.rakuten.co.jp/services/api/Kobo/EbookSearch/20170426`／ジャンル検索 `https://openapi.rakuten.co.jp/services/api/Kobo/GenreSearch/20131010`（**両 API でバージョン番号が異なる点に注意**——電子書籍検索 20170426・ジャンル検索 20131010）。配置直前に公式でバージョン廃止告知が出ていないか最終確認のみ
2. ~~`applicationId` / `accessKey` の渡し方~~ → **解決済（2026-06 公式）**: `applicationId` はクエリ必須、`accessKey` はヘッダ・クエリどちらでも可（本サーバーはクエリで統一）。両方必須
3. レスポンスの `Content-Type`（JSON 以外を返すケースがあるか）。`format=json` 指定で JSON 固定の想定だが念のため確認
4. レート制限値 — 上流は**間隔 1.5 秒未満で 429**の実測報告あり（§3.3 参照）。中継側上限（1秒2req）は粗い保険。実運用での 429 多発時は Android 側ペース（`SiteRateLimiter`）を 1.5 秒以上へ調整
